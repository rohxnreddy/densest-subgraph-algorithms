\section{Introduction}

Finding the densest subgraph involves a strict trade-off between mathematical optimality and computational feasibility. We evaluated two distinct methodologies on three undirected networks: Wiki-Vote, Email-Enron, and AS-Skitter. 

First, from \textit{Efficient Algorithms for Densest Subgraph Discovery}, we implemented Algorithm 1 (Exact) and Algorithm 4 (CoreExact). For these exact algorithms, we specifically set $h=3$ to compute the triangular (3-clique) density. This approach shrinks the graph via $k$-core decompositions before applying parametric maximum flow. Second, from \textit{Flowless: Extracting Densest Subgraphs Without Flow Computations}, we implemented Algorithm 1 (GREEDY) and Algorithm 2 (GREEDY++). These algorithms bypass flow computations entirely, relying instead on iterative degree-based peeling.

\subsection*{Algorithmic Preference: Greedy vs. Exact}

For large-scale graphs, greedy approximations are strongly preferred. While exact flow methods guarantee optimality, their worst-case complexity scales poorly, often stalling on dense graphs. Conversely, greedy approaches provide a $1/2$-approximation in near-linear time. Our C++ implementations processed massive datasets like Wiki-Vote efficiently, proving that the memory and speed advantages of greedy peeling heavily outweigh the practical necessity of an exact mathematical optimum.

\section{Flow Algorithm Usage Choice}

To compute the maximum flow for the \textbf{Exact} and \textbf{CoreExact} algorithms detailed in the work on Densest Subgraph Discovery, we implemented Dinic's algorithm rather than the Edmonds-Karp algorithm. This selection is justified by Dinic's strictly superior worst-case time complexity, which is critical for maintaining performance when processing massive, highly connected graphs.

\subsection*{Theoretical Execution Time Estimation}

To contextualize the performance improvement provided by Dinic's algorithm, we formulate an estimation model for the Edmonds-Karp execution time ($T_{EK}$) based on the recorded execution time of Dinic's algorithm ($T_{Dinic}$). 

By analyzing the ratio of their worst-case theoretical bounds, we can isolate the scaling factor driven by network density:

\begin{equation}
    \text{Ratio} = \frac{\mathcal{O}(V E^2)}{\mathcal{O}(V^2 E)} = \frac{E}{V}
\end{equation}

Assuming the constant factors hidden by the asymptotic notation are roughly equivalent for dense graphs, the estimated execution time for Edmonds-Karp scales proportionally to the average node degree ($E/V$):

\begin{equation}
    T_{EK} \approx T_{Dinic} \times \left( \frac{E}{V} \right)
\end{equation}

Applying this estimation model to as-skitter which consists of $V = 1,696,415$ vertices and $E = 11,095,298$ edges—and using our recorded Dinic's algorithm execution time of $T_{Dinic} = 941$ seconds, we can project the theoretical execution time for Edmonds-Karp:

\begin{equation}
    T_{EK} \approx 941\text{ s} \times \left( \frac{11,095,298}{1,696,415} \right)
\end{equation}

\begin{equation*}
    T_{EK} \approx 941\text{ s} \times 6.54 \approx 6154.5\text{ s}
\end{equation*}

Converting this result yields approximately $1.71$ hours. This estimation demonstrates that Edmonds-Karp would take nearly two hours to process the same graph that Dinic's algorithm solved in roughly 15 minutes. This drastic disparityclearly dictates the necessity of Dinic's algorithm for the dataset.
